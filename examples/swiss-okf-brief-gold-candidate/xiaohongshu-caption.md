# 小红书标题

Google 推出的 OKF，可能是 Agent 知识交付的新信号

# 正文

Google Cloud 最近推了一个 Open Knowledge Format，简称 OKF。这个东西看起来像一个技术规范，但我觉得它真正有意思的地方，是把“组织知识怎么交给 AI Agent”这件事，从产品功能推向了开放格式。

第一条观察：很多 Agent 项目真正卡住的点，常常不是模型能力或工具数量，而是上下文太散。指标定义在文档里，表结构在数据目录里，业务口径在聊天记录里，runbook 又在另一套系统里。Agent 想回答一个具体问题，往往要临时跨系统拼知识。

第二条观察：OKF 的设计很克制。它用普通文件目录表达层级，用 Markdown 写正文，用 YAML frontmatter 放少量可查询字段，再用普通链接表达概念关系。也就是说，它没有试图把知识锁进某个新平台，而是把知识做成普通工具也能读取的文件包。

第三条观察：这种格式对生产方和消费方都更友好。表、指标、数据集、API、playbook、runbook，都可以变成一个个可读、可检索、可版本控制的知识文件。人能打开看，Agent 也能直接解析；可以放进 Git，可以审查变更，也可以被不同工具消费。

第四条观察：Google 这次同时给了规范、生产方、消费方和样例包。这个细节很重要，因为它证明 OKF 已经开始面向真实工具的写入、浏览和验证，而不只停留在格式文档里。

对 Agent 开发者来说，关键价值是知识交付方式变了。过去你可能要给每个 Agent 单独写检索逻辑、适配器和解析规则；如果知识本身可以用一种通用格式表达，Agent 就能先读结构，再逐层展开细节。

OKF 未必会马上成为事实标准，但它释放的信号很明确：AI Agent 要进入真实工作流，光会调用工具还不够，它还需要一种稳定的方式去理解组织知识。未来比拼的可能不只是模型和工具，而是谁能把知识上下文做成可迁移、可审计、可共同维护的基础设施。

你觉得 Agent 时代，组织知识更应该放在知识库里，还是更应该像代码一样管理？
