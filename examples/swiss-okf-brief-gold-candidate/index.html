02 · Context Problem

Agent 真正卡住的，往往不是模型能力

Context

基础模型越来越强，但一到组织内部场景，答案质量常常被“上下文缺失”限制：模型不知道表的含义、指标口径、系统关系和历史变更。

01

知识散落在不同系统里

元数据目录、wiki、共享网盘、代码注释、Notebook、runbook，各自有各自的入口和格式。

02

每个 Agent 都在重复拼上下文

想回答“周活跃用户怎么算”，它要临时跨系统找 schema、指标定义、join 路径和业务语义。

03

知识被锁在最初产出的工具后面

不同厂商有不同 API、SDK、知识图谱 schema，导致知识本身很难迁移、复用和共享。

Open Knowledge FormatContext is the ceiling
03 · Definition

OKF 不是知识库产品，而是一个 格式

Format

Open Knowledge Format 把 LLM-wiki 这种做法标准化：用普通文件目录表达组织知识，让人类可以读，让 AI Agent 也能直接解析。

It is

一个可交换的知识包

每个 Markdown 文件代表一个概念，比如表、指标、数据集、API、playbook 或 runbook。文件路径就是这个概念的身份。

It is not

不是又一个知识服务

它不要求新运行时、不要求 SDK、不绑定云厂商。你可以把它放进 Git、打包成 tarball，或者挂载到任意文件系统。

05 · Design Principles

它只强约束最少的东西

Type

OKF 的重点不是替你规定完整内容模型，而是定义一个互操作面：不同生产方写出来的知识，能被不同消费方读懂。

01

最小化强约束

每个概念只强制需要一个 `type` 字段。其他字段、章节和类型体系，都由生产方自行扩展。

02

生产方和消费方解耦

人手写的知识包、流水线导出的知识包、LLM 生成的知识包，都可以被另一个工具或 Agent 消费。

03

格式，不是平台

它不绑定 Google Cloud、BigQuery、某个模型或某个 Agent 框架。格式本身才是契约。

Only required field: typeProducer / consumer independence
06 · Reference Implementations

Google 一起开源了哪些东西？

Ship

这次发布不是只给一份规范。Google 同时放出了生产方、消费方和样例包，用来证明 OKF 可以被真实工具写入、浏览和验证。

Producer

BigQuery enrichment agent

遍历数据集，为每张表和视图草拟概念文档，再补充引用、schema 和 join 路径。

Consumer

静态 HTML graph visualizer

把任意 OKF 文档包变成可浏览的关系图；没有后端，查看时数据也不需要离开页面。

Samples

三份样例文档包

GA4 e-commerce、Stack Overflow、Bitcoin public datasets，作为合规 OKF 的活样本。

07 · Why it matters

对 AI Agent 开发者，它改变的是知识交付方式

Agent

过去你可能要给每个 Agent 单独写检索、适配和解析逻辑。OKF 想把这层变成通用“知识货币”：谁都能产出，谁都能消费。

01

把内部知识放到版本控制里

指标、表、API、runbook 跟代码一起演进，变更可以审查、回滚、追踪。

02

让 Agent 不再临时拼碎片

上下文被整理成可导航的 Markdown 图谱，模型可以先读结构，再逐层展开细节。

03

让知识离开原来的工具也能活

同一份知识包可以被浏览器、搜索索引、数据目录或另一个 Agent 使用。

04

为“Agent 维护知识”留出空间

LLM 擅长更新交叉引用、同步文件、补充说明；人类负责策展和审核。

08 · Closing

这次真正重要的，是 格式本身

Lingua

如果 AI Agent 要长期接管真实工作，它需要的不只是工具调用能力，还需要一层可迁移、可审计、可共同维护的知识上下文。

OKF 的野心，是让这些知识不再困在某个产品里，而是变成不同系统之间都能交换的通用语。

记住

Markdown 是身体

正文仍然保持人类可读，任何编辑器、GitHub、搜索工具都能处理。

记住

YAML 是索引

只把需要查询的结构化字段放在 frontmatter 里，避免把格式做重。

记住

链接是关系

普通 Markdown 链接让文件目录变成概念图谱，而不是一堆孤立文档。
