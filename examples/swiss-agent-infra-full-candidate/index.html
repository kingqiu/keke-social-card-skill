2026 上半年 / Agent 基础设施演进观察

Agent 基础设施
变厚

这组卡看一个核心变化：Agent 从 demo 走向生产时，模型外面的运行时系统正在变得更厚。

这里的运行时不是某个产品名，而是工具、权限、上下文、状态、审计、恢复、评估和人机介入组成的系统外壳。

01

DEPLOYMENT / SHIFT

FROM MODEL TO DEPLOYMENT

从模型
到部署

模型还是底座，但厂商的产品动作已经开始围绕岗位角色、任务状态、协作界面和部署证明展开。

ROLEAgent 不再只服务开发者，而是进入具体岗位流程
FLOW角色、工具和工作流开始被产品化
PROOF任务状态、CI、PR、崩溃重启都要能交代
POINT竞争从单次回答，转向能否进入真实协作系统
FDE / DEPLOYMENT

FORWARD DEPLOYED ENGINEERING

部署
变成产品

FDE 重新变热，说明企业 AI 的瓶颈不只是有没有模型，而是模型怎么进入复杂组织的生产流程。

AUDIT先审计业务流程和权限边界
EVALS再证明 AI 是否真的有效
DEPLOY从最小自主权开始部署
ORG系统要接住 Agent，组织也要接住 Agent
HARNESS / BOUNDARY

HARNESS AS BOUNDARY

Harness
是边界

Harness 不只是提示词和工具的外壳，而是把 Agent 能做什么、不能做什么写进系统里。

DIR目录读写要有范围，不靠模型自觉
CMD危险命令要批准，不能自动放行
NET网络和凭据要隔离，减少爆炸半径
HUMAN失败以后要能停下，交给人兜底
CONNECTORS / ROUTING

API · CLI · MCP

不是站队
是选路由

API、CLI、MCP 不该被当成信仰之争。真正要问的是：Agent 要接哪里，权限怎么管，结果要不要进上下文。

API单点能力清楚时，直接 API 最省事
CLI本地开发场景里，命令行反馈最快
MCP跨客户端和远程系统，需要标准化接入
RULE工具很多时按需发现，别全塞进上下文
CONTEXT / ROUTING

FROM WINDOW TO SCHEDULING

上下文
变调度

上下文能力不只是窗口变大，而是哪些信息常驻、按需加载、压缩，哪些留在工具和文件系统里。

STATIC系统提示、工具和项目规则要稳定，利于缓存
LOAD参考资料和 Skills 应该按需进入上下文
COMPRESS长任务要压缩中间状态，避免污染
TOOLS大量中间结果留给工具处理，别都交给模型读
RUNTIME / PRODUCTION

NOT ONLY FRAMEWORK

需要 Runtime
不只是框架

Demo 跑一次就结束；生产里的 Agent 要长期运行、保存状态、等待审批，并在出错后能查清楚发生了什么。

DURABLE长任务要能持续执行和恢复
MEMORY跨会话状态不能只靠聊天记录
TENANT多用户和多租户需要隔离
TRACE观测、审计和轨迹，是生产系统的底座
LOOP / FEEDBACK

LOOP ENGINEERING

重点不是循环
而是反馈

让模型多跑几轮并不难。真正难的是每一轮之后有没有验证、事件、轨迹和改进机制。

AGENT模型判断下一步、调用工具、观察结果
VALIDATE没有验证的 loop，只是在反复生成
EVENT事件驱动让任务能被中断、恢复和继续
IMPROVE分析轨迹和失败模式，才可能持续改进
ENTERPRISE / DECISION

BEFORE MULTI-AGENT

先分任务
再上 Agent

Agent 变热，不等于所有任务都要 Agent 化。先判断任务是不是简单、路径是否稳定、步骤是否开放。

SIMPLE简单任务：一次模型调用就够，不要过度工程
STABLE路径稳定：Workflow 更便宜、更可控、更好调试
OPEN步骤开放：需要动态判断时，再交给 Agent
MULTI多 Agent 要等真实瓶颈出现，再拆边界清楚的任务
2026 H2 / WATCHLIST

WHAT TO WATCH NEXT

下半年
看什么

最值得跟踪的不是谁又发了漂亮 demo，而是谁能把 Agent 放进组织和真实系统里稳定运行。

DEPLOY部署能力会不会成为厂商竞争核心
HARNESS权限、沙箱、工具调用会不会平台化
CONTEXT上下文裁剪、缓存和 Skills 怎么产品化
RUNTIME谁能留下可复盘的运行轨迹
FDE谁能进入客户现场，把流程和权限接起来
CHOICE企业别只追新模型，要把边界和回滚先设计进去

这些活儿不花哨，但决定了 Agent 能不能从 demo 走到生产。
