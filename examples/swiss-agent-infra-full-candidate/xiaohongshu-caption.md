# 小红书标题

2026 上半年，Agent 基础设施正在变厚

# 正文

我以前看 Agent，也很容易先看模型榜：谁跑分更高，谁的 demo 更顺，谁又做了一个看起来很会干活的助手。

但这半年越看越觉得，Agent 真正难的地方，常常不在“它能不能回答”，而在“它能不能被放进一个真实系统里”。

真实系统里有很多不性感的东西：权限、工具、上下文、状态、审计、失败恢复、人工介入。demo 里可以先绕过去，生产里绕不过去。

所以这组 10 张卡，我没有按厂商新闻来排，而是按一条主线拆：

Agent 从 demo 走向生产时，模型外面的运行时系统正在变厚。

里面重点看这些问题：

1. 主战场为什么从模型能力，转向部署能力  
2. FDE 重新变热，说明部署本身正在产品化  
3. Harness 为什么是边界，不只是工具外壳  
4. API、CLI、MCP 到底该按什么场景选  
5. 上下文管理为什么开始像信息调度  
6. 生产里的 Agent 为什么需要 Runtime  
7. Loop Engineering 的重点为什么是反馈  
8. 企业落地前，为什么要先分任务，再上 Agent  
9. 下半年值得看哪些基础设施能力

我现在看 Agent 产品，会多问一句：它有没有把这些脏活做进去？

如果没有权限边界、上下文分层、运行轨迹和失败后的恢复机制，再漂亮的 demo 也很容易停在演示层。

这也是我觉得这篇值得拆成 10 张卡的原因。它不是在追某一个新概念，而是在把 Agent 落地时经常被忽略的那层东西摊开看：哪些该交给模型，哪些该交给 workflow，哪些必须由 runtime、harness 和组织流程接住。

如果你在做 AI 产品、企业内部工具，或者只是想判断一个 Agent 项目靠不靠谱，这组卡可以当成一个检查清单：别只问“模型聪不聪明”，也问问“系统能不能兜住它”。

你最近看 Agent 产品时，更关心模型能力，还是这些部署和工程化能力？评论区一起聊。
