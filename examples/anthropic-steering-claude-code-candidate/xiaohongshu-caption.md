# 小红书标题

Claude Code 定制，规则该放哪？

# 正文

Anthropic 这篇文章其实在讲一个很实用的问题：想让 Claude Code 更听话、更稳定，指令到底应该放在哪里？

我以前也容易把所有偏好都往 CLAUDE.md 里塞。后来越用越觉得，这样会让上下文变重，而且有些事情本来就不该靠模型“记住”。这篇文章的关键提醒是：不同定制方式，解决的是不同层级的问题。

第一，项目事实和团队约定可以放在 CLAUDE.md，但不要把很长的流程手册全塞进去。它会常驻上下文，越写越贵。

第二，和某些路径、文件类型绑定的约束，更适合放进 Rules。比如只影响 `src/api/**` 的校验规则，就不该在每次写文档时也被加载。

第三，部署、发布、代码审查、生成报告这类程序性流程，更适合做成 Skills。它们按需加载，能把步骤、脚本和资源组织在一起。

第四，日志分析、深度搜索、依赖审计这种会产生大量中间结果的任务，适合交给 Subagents，让主会话保持干净。

第五，必须确定执行的事情，比如保存前格式化、执行前拦截危险命令、完成后发通知，应该用 Hooks。防护栏不要只靠一句“永远不要”。

最后，Output Styles 和 append-system-prompt 要谨慎用。前者权重高但容易改变默认行为，后者适合临时追加标准。使用它们之前，最好先判断自己改的是交互风格、系统提示，还是一次性的任务约束。

我的理解是：Claude Code 定制的核心，是把规则放到成本、范围和可靠性都匹配的位置。常驻的东西越少，按需加载越清楚，确定性控制越交给代码，Claude Code 才越容易长期稳定。

你会优先把团队约定放在 CLAUDE.md、Rules、Skills，还是 Hooks？欢迎在评论区聊聊你的做法。
