# Aesthetic Standards

This file defines the publishing-quality visual bar for 克克社交卡片. Use it before designing or revising any social card set.

## Priority Order

When improving this Skill, optimize in this order:

1. **Template inheritance first**: preserve the source template grammar before judging beauty or adding new ideas.
2. **System standards second**: define what good output means before changing examples.
3. **Visual systems third**: improve Rednote Native, Proof Lab, Lookbook Journal, Editorial E-ink, and Swiss System as reusable systems.
4. **Golden examples fourth**: create representative examples that prove the systems work.
5. **Individual demos last**: use demos to verify the system, not as one-off hand-designed rescue work.

Do not keep iterating on a single demo if the underlying visual system is weak.

If a card looks publishable but no longer follows the inherited template grammar, treat it as a new visual system candidate, not as a valid upgrade of the original system.

## Publishing-Level Bar

A card set is publishable only when it satisfies all of these:

- It has one strong first-impression idea within 1 second.
- It looks native to the target platform, not like a web dashboard screenshot.
- It has a clear evidence/result area, not only decorative illustration.
- The visual hierarchy is obvious at 25% thumbnail size.
- The composition feels intentional: no accidental blank zones, floaty cards, or arbitrary grids.
- The palette has contrast and warmth without becoming one-note beige, one-note blue, or generic pastel.
- Type feels designed: strong title, readable body, stable line breaks, no cramped or over-shrunk text.
- Female/lifestyle-oriented work should feel specific, useful, and tasteful; avoid cute-but-cheap stickers, childish vector people, and plastic SaaS UI.
- AI/tool work should feel credible and concrete; avoid fake robot art, abstract glow, and unreadable proof.
- Every page in a set must feel like the same visual family.

## Blocking Visual Issues

Reject or redesign when any appear:

- A page looks like an engineering demo, wireframe, or admin dashboard.
- The main proof/result is smaller than surrounding decoration.
- Cards float without a real composition logic.
- Cropped images lose the subject, face, product, outfit, or key UI.
- Large blank regions are unintentional or look like missing content.
- Text repeats the obvious instead of adding a useful reason to continue.
- The design depends on one good cover while later pages collapse in quality.
- A generated or illustrated person looks crude, stiff, childish, or out of category.
- The page looks made by copying an old output instead of applying the visual system.

## Platform-Specific Bar

### Xiaohongshu / Rednote

- First card must sell the tap: pain point, result, or emotional promise.
- Page 2 should quickly explain why the promise is credible.
- Later pages should show proof, process, result, comparison, checklist, or saved value.
- Avoid over-clean corporate grids. Xiaohongshu favors useful density, texture, proof, and personal relevance.
- For female users, prioritize scenario language: 通勤, 约会, 见客户, 显瘦, 有气色, 不用力, 能走路.

This Skill does not generate WeChat Official Account covers or forwarding images. When the source is a WeChat article, adapt the idea into Xiaohongshu card-style images instead.

## Image And Asset Rules

- If a route needs proof, use real screenshots, real output, user-provided assets, or generated assets that look like the target result.
- Do not use low-quality CSS/SVG stand-ins as final proof unless the task is explicitly a wireframe or template demo.
- If AI-generated full-card imagery is used as a visual reference or source asset, record it in `SOURCES.md` and explain how it is used.
- For deterministic HTML cards, prefer composing real visual assets inside HTML over hand-drawing complex lifestyle subjects from scratch.

## Self-Review Before Delivery

Before final delivery, inspect rendered images manually and answer:

1. Would this look embarrassing beside a strong Xiaohongshu post in the same category?
2. Is the weakest page still acceptable, or did only the cover pass?
3. Is the evidence/result large enough to inspect?
4. Does each page have a reason to exist?
5. Is the visual language reusable for the next project?

If any answer is weak, continue improving the system or example before calling it done.
